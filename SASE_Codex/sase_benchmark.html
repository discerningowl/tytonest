Architectural Framing
Understanding Single-Pass vs. Stitched SASE before evaluating any vendor

The Most Important Architectural Decision in SASE

Before comparing vendors on features, buyers need to understand the single most consequential architectural split in the market: Single-Pass vs. Stitched (Multi-Engine) SASE. Every vendor's strengths and weaknesses flow from which side of this divide they sit on.

SINGLE-PASS

Native / Converged Architecture

Traffic is inspected once by a unified policy engine that applies SWG, CASB, ZTNA, FWaaS, and DLP checks simultaneously. Security and networking logic share the same data plane.

  • Lower latency — no chaining of proxy hops
  • Consistent policy — one rule set, no gaps between engines
  • Simpler operations — single console, single data model
  • Better DLP — full context is available in one pass
Cato Networks Netskope (NewEdge) Zscaler (ZIA/ZPA)
STITCHED

Integrated / Multi-Engine Architecture

Capabilities are bolted together from acquired products or partner integrations. Traffic may traverse multiple engines, each adding latency and creating potential for policy inconsistency.

  • Higher latency — multiple inspection hops
  • Policy gaps at integration seams
  • Heavier operational overhead
  • Often superior depth in individual components
Palo Alto (Prisma + SCM) Cisco (SASE) Fortinet (FortiSASE)
Single-Pass vs Stitched: Imagine a security checkpoint at an airport. Single-Pass is one officer who checks your ID, your bag, and your boarding pass simultaneously. Stitched is three separate lines — ID check, then bag scan, then boarding pass — each with its own queue. You get to the gate either way, but the single officer is faster and has the full picture. The tradeoff: that one officer needs to be very good at all three jobs.

Why This Matters for the Benchmark

Stitched vendors can still score highly — especially where deep per-component capability matters (e.g., Palo Alto's threat prevention depth). But single-pass vendors will score better on operational simplicity, DLP accuracy, and latency SLAs. This document scores both dimensions explicitly so buyers can weight them according to their own priorities.

Pillar 1 — Remote Access / ZTNA
Replacing legacy VPN with identity-based, per-application, continuous-trust access

What ZTNA Is (and Isn't)

Legacy VPN grants network access — once authenticated, a user can reach anything on the subnet. ZTNA grants application access — the user is authorized to a specific app, on a specific device, in a specific context, for a specific session. Trust is never implicit; it is continuously re-evaluated.

ZTNA is built on ten operating principles (see source materials), but for benchmarking purposes, five are differentiating in 2026:

1. Device Posture Assessment

Does the solution check device health (OS patch level, EDR status, certificate presence, disk encryption) at connection time and continuously throughout the session? Best-in-class solutions integrate with major EDR platforms (CrowdStrike, SentinelOne, Microsoft Defender) to pull real-time posture signals — not just a one-time snapshot at login.

2. App-Level Micro-Segmentation

Can access policies be enforced at the individual application or API endpoint level — not just at the network segment level? Best-in-class solutions support private app connectors that publish apps without requiring inbound firewall rules, making the app invisible to the internet entirely.

3. Global Latency & PoP Distribution

ZTNA is only viable as a VPN replacement if latency is competitive. Best-in-class requires <150ms RTT for 95% of global users to the nearest PoP, a private backbone (not hairpin via public internet), and active/active PoP failover with sub-5-second reconvergence.

4. Agentless / Browser-Based Access

For contractors, BYOD users, and VDI reduction, best-in-class solutions provide browser-isolated access that delivers a published app without installing a client. This is now a table-stakes capability in 2026; the differentiator is how well it handles thick-client apps (RDP, SSH, thick ERP).

5. Enterprise Browser

A managed, policy-enforced browser deployed as the primary access vehicle — distinct from agentless RBI. Enterprise browsers (Island, Talon/Palo Alto, Chrome Enterprise) enforce DLP, ZTNA policy, and data controls at the browser layer itself, eliminating the need for network-side inspection for managed devices. In 2026 this is an emerging but fast-moving category: some SASE vendors are building native enterprise browser capability, others are partnering, and some have no answer. It matters because it changes the architecture conversation — a well-deployed enterprise browser can reduce the scope of what the SASE proxy needs to inspect.

Pillar 2 — Security Service Edge (SSE)
Converging SWG, CASB, FWaaS, DLP, SSL Decryption, RBI, and AI/GenAI Protection into a unified cloud security stack

SSE: Seven Technologies, One Architecture Problem

SSE is the security half of SASE. It consolidates historically separate security products — SWG, CASB, FWaaS, DLP, and SSL/TLS decryption — into a cloud-delivered service. The consolidation only delivers value if all components share a unified policy engine and data store; otherwise it is a billing bundle, not an architecture.

SWG

Secure Web Gateway

Inspects all web and internet traffic. Best-in-class in 2026 goes well beyond URL filtering:

  • Inline TLS 1.3 decryption at scale without latency penalty
  • AI-categorized URL/content classification (not just static databases)
  • Remote Browser Isolation (RBI) for risky or uncategorized sites
  • DNS security integrated (not bolted on)
  • Zero-day malware sandboxing with <30s verdict time
CASB

Cloud Access Security Broker

Governs access to sanctioned and unsanctioned SaaS. Key differentiation:

  • Inline + API-based dual mode (inline for real-time, API for historical scan)
  • Shadow IT discovery with business risk scoring
  • Granular SaaS control (e.g., allow Google Drive upload but block external share)
  • Tenant-to-tenant controls (block personal Google to corporate Google data movement)
  • App-to-app OAuth grant visibility
FWaaS

Firewall as a Service

Cloud-native L3–L7 firewall replacing data center perimeter firewalls. Benchmarks:

  • Full L7 application awareness (not just port/protocol)
  • IPS/IDS with real-time threat intelligence updates
  • Egress controls for branch and cloud workloads
  • East-west traffic inspection in cloud environments
  • Policy consistency between branch and remote user (no split policy model)
DLP

Data Loss Prevention

DLP is the most differentiated capability in SSE in 2026. Basic DLP uses regex patterns. Best-in-class uses ML:

  • ML-based content classification (not just regex/fingerprint)
  • Exact Data Matching (EDM) for structured PII datasets
  • Optical Character Recognition (OCR) for images and screenshots
  • Document fingerprinting across cloud and endpoint
  • Unified DLP policy across SWG, CASB, and ZTNA (one rule = everywhere)
  • Generative AI data protection — detect prompt injection and LLM data exfil
SSL/TLS

SSL/TLS Decryption

SSL inspection is the prerequisite for almost everything else in SSE. Without it, SWG, DLP, and threat detection are operating blind on the majority of traffic. It deserves treatment as a first-class capability, not a checkbox:

  • Full TLS 1.3 decryption without performance degradation at scale
  • Certificate management — forward proxy CA deployment, pinning bypass for known apps (O365, Salesforce)
  • Policy-based decryption exemptions (banking, healthcare, personal categories) with audit trail
  • Inspection at the PoP — not backhauled to a central engine
  • Visibility into encrypted C2 traffic and malware exfiltration channels

Key differentiator: Palo Alto (Prisma Access) carries NGFW-grade decryption into the cloud proxy. Vendors with bolt-on SSL inspection often show latency cliffs at scale or incomplete TLS 1.3 support.

RBI

Remote Browser Isolation (RBI)

RBI is architecturally distinct from SWG URL filtering — it does not block bad sites, it makes all sites safe by never executing web content on the endpoint. Two rendering models exist in 2026:

  • Pixel-push: Server fully renders the page and streams a pixel image to the client. Zero DOM, zero JavaScript, zero active content reaches the endpoint. Maximum isolation, some latency overhead.
  • DOM reconstruction: Server fetches the page, strips dangerous content, and reconstructs a sanitized DOM in the client browser. Lower latency, minor fidelity trade-offs for complex web applications.

Vendor split: Zscaler (native pixel-push), Cloudflare (native, low-latency), Menlo (RBI-first, not in Big Five), Palo Alto (native DOM isolation), Netskope (native pixel-rendering). Cato: RBI available as add-on. All differ on whether RBI shares a policy engine with SWG — integration depth is the differentiator, not just availability.

AI / GENAI

AI/GenAI Data Protection

The fastest-growing SSE requirement in 2026. Employees are pasting PII, source code, M&A details, and regulated data into public LLMs at a rate that outpaces most security policies. Best-in-class goes beyond blocking ChatGPT by category:

  • Inline DLP inspection of LLM prompts and responses — not just app-level block/allow
  • Per-model controls (block code uploads to public models; allow general queries)
  • Real-time user coaching when sensitive content is detected before submission
  • Visibility into 500+ AI tools across the org — shadow AI discovery
  • Prompt injection detection — prevent LLMs from being weaponized against the user

Vendor split: Netskope leads with content-aware prompt inspection tied to its unified DLP engine. Palo Alto AI Access Security provides inline inspection per AI app. Zscaler enforces at the app category level — meaningful gap for regulated industries.

DLP Is Not One Thing — Vendor Philosophy Determines Fit: Each vendor implements DLP from a different design center. There is no universal "best" — the right choice depends on what data problem you are actually solving. The DLP Deep Dive will map each vendor's DLP approach to specific use cases.
Netskope — Data-Centric by Design

DLP is the product's core identity, not a feature layer. ML classification, EDM, OCR, and document fingerprinting share a single data store across inline and API modes. Policy written once applies everywhere — SWG, CASB, and private apps. Best fit: Organizations where data visibility and classification accuracy are the primary security requirement (financial services, legal, life sciences).

Zscaler — Proxy-Enforced DLP

DLP is enforced inline through the ZIA proxy for internet traffic and ZPA for private app access. Strong for web and cloud channel coverage. Architecture note: ZIA and ZPA are distinct products that share policy through integration rather than a native unified engine — worth validating in the SSE deep dive. Best fit: Organizations prioritizing ZTNA architecture and digital experience monitoring where DLP is a compliance requirement rather than a primary differentiator.

Palo Alto (Prisma + SCM) — Threat-First DLP

DLP capability is broad across the Prisma platform and benefits from WildFire threat intelligence integration. The design philosophy prioritizes threat prevention depth (ML-NGFW, sandboxing) over data classification fidelity. DLP policy is managed through Strata Cloud Manager alongside the broader security estate. Best fit: Organizations that need DLP as part of a comprehensive security posture where NGFW-grade threat prevention is the anchor requirement.

Cato & Cloudflare — Operational DLP

Both vendors provide DLP coverage sufficient for standard compliance requirements (PCI, HIPAA pattern matching, common PII templates). Neither is designed for deep customization or complex data classification workflows. Best fit: Organizations where operational simplicity and consolidated networking+security matter more than DLP depth — mid-market, lean IT teams, SaaS-first environments.

RBI Architecture — Why Rendering Model Matters

Pixel-Push (Server-Side Rendering)

The server renders the full page and streams a compressed pixel image to the client. The endpoint receives no DOM, no JavaScript, no active content — only a visual representation. Any zero-day exploit in the page code fires on an isolated cloud container and dies there. Latency cost: 30–80ms typical overhead from pixel encoding and streaming. Use when: high-risk user populations, uncategorized sites, financial services regulatory requirements.

DOM Reconstruction (Clientless Rendering)

The server fetches the page, strips scripts and active content, and passes a sanitized DOM to the client browser for local rendering. The client renders its own view of the sanitized page. Latency cost: 5–20ms typical overhead — materially lower than pixel-push. Fidelity trade-off: complex JavaScript-heavy applications (SPAs, web apps with heavy client-side logic) may render incorrectly or incompletely. Use when: broad deployment across all web browsing, latency-sensitive users, or as the default RBI posture with pixel-push reserved for highest-risk sites.

Integration with SWG Policy Is the Key Differentiator: Having RBI is table stakes in 2026. Having RBI that shares a policy engine with SWG is not. Best-in-class: a SWG rule that says "isolate all uncategorized sites" automatically routes matching traffic through RBI — no separate policy object, no separate console. In stitched architectures, RBI and SWG are separate products with separate policies that must be manually kept in sync. Gaps in synchronization mean isolation can be bypassed by any category that falls outside the RBI policy scope.

Pillar 3 — SD-WAN
Physical and virtual edge connectivity — the "A" in SASE

SD-WAN: The Network Foundation of SASE

SD-WAN is often undersold as "just routing" in SASE conversations dominated by security. This is a mistake. A vendor's SD-WAN architecture determines WAN performance, branch reliability, and the quality of the middle-mile path between users and cloud workloads. In 2026, the key split is between vendors with native private backbones vs. those relying on public internet routing with optimization.

THIN EDGE

Cloud-Heavy Architecture

Minimal on-premises hardware. Security, routing, and optimization are handled in the cloud or at the nearest PoP. Best suited for organizations with:

  • Small branches or retail locations
  • Heavy SaaS adoption (M365, Salesforce, Workday)
  • Low tolerance for on-site IT complexity
  • Priority on fast deployment and managed service

Examples: Cato Socket, Cloudflare One Appliance (formerly Magic WAN Connector), Zscaler Branch Connector, Netskope SASE Gateway

THICK EDGE

Branch-Heavy Architecture

Significant processing at the branch: local security enforcement, application optimization, offline resilience. Best suited for:

  • Manufacturing or retail with latency-sensitive local apps
  • Environments with intermittent WAN (remote sites)
  • Regulated industries requiring local data inspection
  • Large branches needing local LAN-side security

Examples: Palo Alto Prisma SD-WAN (ex-CloudGenix), Fortinet SD-WAN, Versa

Pillar 4 — AIOps & Autonomous Operations
Using GenAI and ML to reduce operational burden and accelerate threat response

AIOps: From Monitoring to Autonomous Action

AIOps in SASE covers two distinct problems: network operations (diagnose why a user in Singapore has bad latency to Salesforce) and security operations (correlate 47 low-confidence signals into one high-confidence threat incident). Best-in-class vendors in 2026 are moving beyond dashboards into autonomous remediation — where the system proposes or executes corrective actions without a human ticket.

GenAI Policy Authoring

Natural language to policy rule translation. An admin types "block all uploads to personal cloud storage from unmanaged devices during work hours" and the system generates the policy object. Best-in-class includes policy simulation (what traffic would this rule affect?) before deployment.

Anomaly Detection & UEBA

User and Entity Behavior Analytics that baselines normal behavior per user, per app, per time-of-day — and alerts on deviations. Best-in-class uses multi-dimensional models, not single-threshold rules. Detects insider threat and credential abuse that signature-based tools miss.

Middle-Mile Diagnostics

Automated root-cause analysis for latency complaints: is the problem the ISP, the PoP, the application, or the device? Best-in-class provides a single "path health score" per session with automatic ISP fault attribution — replacing manual traceroutes and MTR analysis.

Incident Correlation & SOAR

Cross-product event correlation that links a SWG alert, a CASB anomaly, and a ZTNA posture failure into one incident ticket. Best-in-class auto-enriches with threat intelligence and suggests containment actions (e.g., "isolate this user's access pending review").

Pillar 5 — Sovereignty-by-Design
Data residency, PoP-level localization, and regulatory compliance in 2026

Why Sovereignty Is Now a Scored SASE Criterion

Data residency requirements have tightened significantly entering 2026. GDPR enforcement has expanded, Brazil's LGPD is fully active, India's DPDP Act is in force, and the EU AI Act's data governance provisions are being applied to cloud security services. For global organizations, a SASE vendor's ability to guarantee data stays within a jurisdiction — at the PoP level, not just contractually — is now a procurement requirement, not a checkbox.

The Critical Distinction: "Data residency" claimed in a contract is not the same as data residency enforced in architecture. Best-in-class vendors can demonstrate that logs, metadata, and DLP-inspected content never physically leave the designated region — verified through architecture documentation, third-party audit, or sovereign PoP design.

Buyer Persona Weighting
Not all criteria matter equally to all buyers — here's how to weight the rubric by organizational profile

Three Archetypes, Three Weighting Profiles

The same vendor can be the right answer for one organization and the wrong answer for another, depending on what they value most. Vendor scores will be filtered through each of these three personas.

PERSONA A

Lean IT / Mid-Market

500–3,000 employees · Small security team · SaaS-first

  • Highest weight: Operational simplicity, single console
  • Highest weight: ZTP and fast deployment
  • Highest weight: Managed service / co-managed option
  • Lower weight: Deep DLP customization
  • Lower weight: Sovereign PoP options
Best-fit vendors likely: Cato Networks, Cloudflare
PERSONA B

Global Security Ops

5,000+ employees · Dedicated SOC · Regulated industry

  • Highest weight: DLP depth and accuracy
  • Highest weight: UEBA and threat detection
  • Highest weight: SIEM/SOAR integration
  • Highest weight: Sovereignty and certifications
  • Lower weight: Ease of initial setup
Best-fit vendors likely: Netskope (DLP/CASB-primary), Palo Alto (threat prevention-primary), Zscaler (ZTNA/DEM-primary)
PERSONA C

Global Network Ops / SD-WAN First

Multi-site enterprise · MPLS replacement · Performance-driven

  • Highest weight: Private backbone quality
  • Highest weight: App-aware path steering
  • Highest weight: Branch hardware flexibility
  • Highest weight: Global PoP coverage
  • Lower weight: Advanced DLP
Best-fit vendors likely: Cato, Aryaka, Palo Alto SD-WAN
Note on Emerging Vendors: Aryaka, Graphiant, Nile, and Island will be evaluated in the Emerging Vendors section. Aryaka competes on Pillars 3+4 (SD-WAN + managed ops). Graphiant competes on Pillar 3 only (private connectivity). Nile competes on a new category: Zero Trust campus LAN, which extends ZTNA principles inward to the physical building. Island is a full Chromium fork enterprise browser enforcing Zero Trust policy at the browser layer — distinct from proxy-side inspection. None are wholesale SASE replacements. Full analysis in Emerging Vendors.
